What · beat 10:15 – 0:50

What is PR rot?

A pull request is a race against your main branch. While it waits for review, main keeps moving — and past a point the PR can no longer merge: conflicts, broken checks. The finished work inside it starts to rot. Someone has to pay the merge tax: rebase it, fix it, or lose it.

P(stuck)  ≈  time open × main churn × diff size   — physics, not process failure. Every concurrent team pays it.

Every rotted PR is a double loss — engineering you already paid for, and an improvement your codebase never gets.

How · beat 63:10 – 3:45

The architecture, in five decisions.

01

Control plane vs. agent plane. The loop — detect, classify, dispatch, verify — is a deterministic state machine in plain code. The agent is confined to the one stage that needs judgment — resolving conflicts, repairing CI. A failed, stuck, or lying session becomes a failed work item — never a wedged system.orchestrator.py — no model call anywhere in the control path

02

Level-triggered, not event-dependent. Discovery is sweep-only: each cycle re-derives the blocked set from repo state, so dropped events cost nothing — there are none to drop. The single webhook is the label, the one path where a human is waiting.main.py · run_sweep() — reconciliation, the Kubernetes argument

03

Held to the same bar as any teammate. A hard budget per session; work we can't prove is mechanical is never assigned (the 137 review-bound PRs); and results count only after GitHub confirms them — the discipline we already apply to humans. Worst case: one bad branch, recoverable from git history, still behind human review.max_acu_limit · classify() → return None · _verify_unblocked()

04

Never interrupt a human. A PR pushed to in the last 3 days is work in progress, not rot — the sweep won't touch it until it has gone quiet. Rot takes weeks; the teammate can wait days. The label overrides: a human asking is consent.min_quiet_days · dispatch(force=True) on label

05

Every action leaves a durable record. The tracking issue is the work item, the spend gate, and the audit trail; every state transition is timestamped. The dashboard is a view of the architecture, not a bolt-on.store.py — state machine + event log behind the metrics

Why · beat 73:45 – 4:15

A teammate, not a helper.

Automated — nobody drives it

Copilots assume a developer in the loop. Here there is no loop: the event fires, a session spawns in its own environment, works, reports back, escalates when stuck. The whole integration is one HTTP call — POST /sessions with a prompt, budget, and structured-output contract.

Customizable — the workflow is ours

This isn't a “fix my PRs” button we adopted. We measured a specific problem and built a specific flow — our detection rules, our classifier, our verification, our guardrails — in hours, not a quarter. Devin is the engine; the workflow is the product.
